The module tree contains generic `mediatek-cpufreq-hw.ko`, `mtk-svs.ko`,
`auxadc_thermal.ko`, and `mt6577_auxadc.ko`. Their presence is not a Gemini
consumer binding: no MT6797 cpufreq driver or SVS/OPP consumer is selected by
the board DT, and the thermal/AUXADC nodes remain disabled.

Device-tree boundary
--------------------

The packaged DTB has one PSCI node and ten CPU nodes. It has zero
`operating-points-v2`/OPP-table matches and zero idle-state descriptions. The
only thermal node is `thermal@1100b000` (`mediatek,mt6797-thermal`) and it is
disabled; its paired `adc@11001000` AUXADC node is also disabled. No DVFSP/SPM
consumer, CPU cluster clock handle, regulator consumer, CCI coupling, or
vendor idle-state table is present in the board DTB.

Decision
--------

The current package can describe generic ARM64 CPU topology/PSCI and carries
generic PM/thermal building blocks, but it deliberately does not make a
frequency, voltage, thermal-trip, deep-idle, or suspend transition on Gemini.
The vendor MT6797 path still requires efuse/PTP calibration, external-buck and
SRAM tracking, PLL/mux ownership, CCI coupling, thermal-bank mapping, and SPM/
firmware state contracts. Keep those consumers disabled; do not infer support
from generic module presence or the `mem`/PSCI configuration alone.

frequency_voltage_transition=not_attempted
thermal_probe=not_attempted
deep_idle=not_attempted
suspend=not_attempted
hardware_write=none
