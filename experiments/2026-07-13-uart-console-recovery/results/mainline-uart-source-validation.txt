The generated DTB contains the expected UART0 contract: one UART register
window at `0x11002000`, one GIC SPI interrupt, two clocks named `baud` and
`bus`, the MT6797 plus MT6577 fallback compatibles, and the `serial0` chosen
path at 921600 8N1. The focused binding target passed using the existing build
output. The two checkpatch warnings are the existing commit-message line
length and MAINTAINERS reminder; neither is a source error.

Resource boundary audit (current source): the vendor UART0 node has a 0x1000
UART bank plus AP-DMA TX/RX windows and IRQs 91/108/109. The Linux binding and
board node deliberately retain only a 0x400 UART bank and SPI 91. `8250_mtk`
uses only low MediaTek extension offsets for the PIO path and explicitly turns
DMA off when the port is a console, so the smaller resource is sufficient for
the intended first boot. This does not validate vendor DMA; it records the
exact point at which a future DMA investigation must reopen the DT contract.

The current source audit also inspected `uart0_pins_a` in the Linux MT6797
SoC DTS. It contains only the UTXD0/URXD0 pinmux entries and no
`input-enable`, bias, pull, or drive-strength properties. This is compatible
with the current MT6797 pinctrl implementation and avoids the unsupported
pinconf maps that were removed from the active eMMC groups in patch 0071.
The active UART0 board declaration therefore remains a pinmux-only PIO
contract; no UART source patch is needed for this issue.

Unprivileged live follow-up (2026-07-13): `/proc/cmdline` was unreadable and
the live `/sys/firmware/devicetree/base/chosen/stdout-path` node was absent.
No sudo escalation or device mutation was attempted. The downstream
`console=ttyMT0` observation therefore remains evidence from the earlier
owner-authorized baseline, not a new claim about the current boot.

Decision: reuse Linux 7.1.3 `8250_mtk` with the existing MT6797 compatible,
standard `serial0` stdout path, and PIO console. Do not copy the vendor
`ttyMT*` ABI or enable AP-DMA until the bootloader console token and DMA
resource contract are independently validated.
