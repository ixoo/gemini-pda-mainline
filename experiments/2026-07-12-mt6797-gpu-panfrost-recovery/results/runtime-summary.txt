Static source and mainline comparison
- Vendor `mt_gpufreq` programs MFGPLL and parking/universal PLL paths, selects
  efuse speed bins, and couples VGPU/PMIC and thermal/PBM limits to frequency.
- Its source selects `VGPU_SET_BY_EXTIC`, checks RT5735 product ID 0x10, and
  uses VSEL registers 0x10/0x11 with bit 7 as enable; this is not a proven
  FAN53555-compatible regulator contract.
- The pinned public vendor tree contains generic Arm Mali Midgard/Kbase
  `mali-r12p0` plus the configured `mali-r12p1` MT6797 platform sources under
  `platform/mt6797/`. The ELF correlates with the source-level hooks and
  sequencing; see [mali-vendor-analysis.txt](mali-vendor-analysis.txt).
- Vendor DT lists MFG main/VCG, async/MFG and four MFG-core domains, 52 MHz and
  universal-PLL muxes, GPU DVFS/SPM, GPU-PM I2C, and AP-DMA clocks.
- Linux 7.1.3 Panfrost already matches generic arm,mali-t860 and arm,mali-t880
  and discovers the 0x0880 GPU model from hardware registers.
- Linux 7.1.3 MT6797 SCPSYS currently instantiates mfg_async but not the vendor
  MFG/four-core domain set. Top/infrastructure CCF IDs for MFG/MFGPLL/MFG-VCG/
  GPU-PM exist, but the vendor `mfgsys`/G3D gate provider (`MFG_BG3D`) is not
  yet described by the mainline MT6797 DTS.
- The recovered M4U port table has no GPU client; do not add a GPU `iommus`
  property without contrary evidence.

Conclusion: Panfrost core-model reuse is supported by source and live ID, but
platform power/clock/reset/OPP integration remains unproven. No GPU node was
enabled and no GPU policy or frequency operation was attempted.
