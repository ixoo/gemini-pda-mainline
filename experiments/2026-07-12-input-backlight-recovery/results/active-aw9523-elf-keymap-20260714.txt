## Active ELF anchors

The active boot payload was extracted from the captured Android boot image at
offset `2048`, length `8429825` bytes, then reconstructed with
`vmlinux-to-elf` in the VM. Its embedded Linux build string matches the live
device's `/proc/version` exactly. The active image is distinct from the later
May 2019 installed modular-kernel package; that package's ELF is not used for
this conclusion.

The active `aw9523_i2c_probe()` capability sequence contains `0x7d`
(`KEY_LEFTMETA`) and `0xf0` (`KEY_UNKNOWN`), and contains no `0x1d0`
(`KEY_FN`). The probe stores a 56-entry keymap pointer at the driver state
offset used by the event worker. The recovered table has 28-byte entries with
the keycode at byte offset 12. The `META-L` record is entry 28, which is
`(row=4,col=3)` in the vendor's column-major table—the physical position that
the retained source checkout labels `FN`. The four `NULL` records at
`(row=7,col=3..6)` retain keycode `0xf0`.

## Decision

The exact active boot binary is stronger provenance than the later retained
source checkout for the disputed keycode. The mainline disabled DT candidate
should use `KEY_LEFTMETA` at `(4,3)` to match the active image, while the
retained source-derived table remains evidence of the source/build discrepancy.
An owner-assisted evdev press/release trace is still required to verify the
physical legend, modifier behavior, and userspace function layer before
enabling the consumer; it is no longer necessary to choose between `KEY_FN`
and `KEY_LEFTMETA` for the candidate map.
