Required owner-assisted test, only after a recoverable mainline boot path exists:

1. Keep touch, display, USB host/VBUS, charger, and other optional consumers
   disabled; enable only the already-described AW9523 and matrix-keypad nodes.
2. Capture the input device capabilities and interrupt counters before any
   key press. Press each assigned key once, then test the modifier pairs,
   arrows, function layer, and four-key rollover. Record evdev key codes and
   whether any ghost or stuck key appears. In particular, verify that the
   physical `(row=4,col=3)` position emits `KEY_LEFTMETA`, as recovered from
   the exact active boot ELF, and that the four `KEY_UNKNOWN` positions do not
   create unintended key events.
3. Repeat with the lid closed/open and across suspend/resume only if recovery
   remains available. Check that the parent EINT polarity, GPIO58 reset level,
   debounce, wake behavior, and userspace XKB layer agree with the evidence.
   Capture monotonic timestamps for the parent IRQ, first event, release, and
   any repeated events so the vendor 1-ms/10-ms cadence is not confused with
   Linux `debounce-delay-ms`, `col-scan-delay-us`, or
   `all-cols-on-delay-us`.
4. Do not promote the candidate to enabled-by-default until the one-key and
   modifier results are reproducible on the named device.

This gate is intentionally a plan, not a claim that transitions were
stimulated. The current capture and keymap validator establish topology and
source agreement but cannot establish row/column polarity, debounce/settling,
rollover, LED behavior, timing equivalence, or wake semantics.
