## Source anchors

Linux `matrix_keypad_build_keymap()` allocates the expanded keycode array with
zero-initializing `devm_kcalloc()` (`drivers/input/matrix-keymap.c:158`).  It
then assigns only entries present in `linux,keymap` and advertises each mapped
code (`:38-39`), finally clearing `KEY_RESERVED` from the input capability
bitmap (`:189`).  The input-code header defines `KEY_RESERVED` as `0` and
`KEY_UNKNOWN` as `240`.

The GPIO matrix scanner emits `MSC_SCAN` for every changed physical coordinate,
looks up `keycodes[code]`, and calls `input_report_key()`
(`drivers/input/keyboard/matrix_keypad.c:163-166`).  It advertises the
`MSC_SCAN` capability at `:453`.  The input core passes an `EV_KEY` only when
the keycode is present in `dev->keybit` (`drivers/input/input.c:236`), and
explicitly documents that `KEY_RESERVED` is not transmitted to userspace
(`:2354`).

## Interpretation

The vendor source table contains four `KEY_UNKNOWN` entries at row 7, columns
3–6.  Those entries would be real `EV_KEY/KEY_UNKNOWN` events if copied into a
mainline `linux,keymap`, because `matrix_keypad_map_key()` sets bit 240.  The
candidate instead omits them.  Linux therefore leaves those expanded slots at
`KEY_RESERVED`; a transition can still expose its physical matrix scan code as
`MSC_SCAN`, but its `EV_KEY` event is dropped by the input core.

This is a deliberate safety boundary, not proof that the four contacts are
unpopulated.  Keep the four positions omitted until an owner-assisted evdev
capture establishes whether any physical key generates them.  If a position is
populated, add its verified keycode to the DT map; do not add `KEY_UNKNOWN`
merely to reproduce the vendor capability bitmap.  The existing keymap
validator correctly treats the four coordinates as explicit source-table
spares and requires 52 assigned `MATRIX_KEY()` entries.
