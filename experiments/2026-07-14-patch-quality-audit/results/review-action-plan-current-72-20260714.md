## Submission blockers

The provenance audit finds 32 patch-level blockers that checkpatch cannot
identify from patch content alone:

- 18 patches use `Gemini PDA Mainline Project <noreply@example.com>` as both
  `From:` and `Signed-off-by:` metadata (`0021`–`0036`, `0057`, and `0057a`).
- 8 patches use an all-zero 40-hex format-patch object ID (`0054`, `0055`,
  `0056`, `0057a`, `0062`, `0063`, `0064`, and `0065`).
- 8 patches lack any `Signed-off-by:` line (listed below).

Placeholder author/sign-off records are not DCO evidence, even when the
checker sees a syntactically present footer. Zero object IDs must be
re-exported from real commits. These provenance blockers are separate from
the functional/static diagnostics below.

The following eight patches have `ERROR:MISSING_SIGN_OFF` and must be
re-exported with a truthful `Signed-off-by:` from each actual author. Do not
add a maintainer's sign-off on another author's behalf.

| `0044` | MT6797 display PWM | Obtain the author's DCO sign-off; split the binding change from implementation before submission |

## Binding and ownership work before posting

The checker reports 14 `DT_SPLIT_BINDING_PATCH` warnings. Separate YAML/schema
changes from driver or board changes, then validate each binding with
`make dt_binding_check` and the relevant Gemini DT with `make dtbs_check`.
The affected areas include the NT36672E panel, display PWM, MFGSYS, RT5735,
FAN49101, FUSB301, AUXADC, LK calibration NVMEM, Panfrost, DPI, SCPSYS, AFE,
T-PHY, MTU3/xHCI, and USB11/MUSB additions.

There are 18 `FILE_PATH_CHANGES` warnings. For every newly added binding or
driver, run `scripts/get_maintainer.pl` against the final split patch and add
or update the appropriate MAINTAINERS entry only when subsystem ownership
requires it. The warning is a review prompt, not permission to invent an
owner.

The four `CONFIG_DESCRIPTION` warnings (MFGSYS, RT5735, FAN49101, FUSB301)
need complete Kconfig help text. The seven `COMMIT_MESSAGE` warnings need a
problem statement, hardware evidence, reuse decision, and test status. The 16
long-log-line warnings should be wrapped to 75 columns after the substance is
reviewed.

## C implementation checks

The check-only diagnostics are small but should be resolved before upstream
submission:

- `0005`: two `OPEN_ENDED_LINE` cases in EINT code.
- `0015`: two `MACRO_ARG_REUSE` cases; inspect side effects before deciding
  whether to rewrite the macros.
- `0047`, `0051`, `0057a`: parenthesis alignment checks.
- `0058`: missing blank line after a declaration.

These are source-review items, not grounds to claim or deny runtime support.

## Static compile coverage

The expanded [W=1/sparse result](series-static-compile-20260714.txt) compiles
all 22 subsystem directories touched by C sources in the series with zero
diagnostics. This raises confidence in the source/build path, but does not
validate DT resource ownership, probe-time writes, firmware handshakes, or
runtime hardware behavior.

## Safety and evidence boundary

The patch series remains a compile/package candidate. No sign-off is inferred,
no undocumented consumer is enabled because of checkpatch output, and no
device write or runtime probe was performed for this review record.

## Exit criteria

1. Re-export affected patches from real commits with truthful author/DCO data.
2. Split binding, driver, and board changes into reviewable patches.
3. Re-run checkpatch, binding/schema checks, sparse/W=1, and the VM build.
4. Keep runtime claims tied to named-device evidence; static cleanliness does
   not promote a disabled or module-only path to hardware support.
