## What the commit demonstrates

The reference commit reports a readable, rotated framebuffer console on the
physical Gemini display using the native MediaTek DRM pipeline. It is not a
simplefb handoff. Its production configuration enables DRM, the MediaTek
display pipeline, framebuffer emulation and fbcon, the MT6797 MIPI DSI PHY,
panel, bias, PWM, I2C, MMSYS clocks, and power domains as built-ins. It disables
simplefb and no longer needs `clk_ignore_unused` once real consumers own the
display clocks.

The reported successful combination includes:

- quiescing an overlay interrupt left enabled by LK before registering the
  Linux IRQ handler, plus rejecting an interrupt before the CRTC has atomic
  state;
- putting DSI into command mode after reset, before sending panel
  initialization transfers;
- MT6797-specific D-PHY `TIMCON0` through `TIMCON3` values recovered from the
  working LK state;
- a corrected MT6797 MIPI TX PLL/bandgap implementation and power sequence;
- video mode, four data lanes, RGB888, and vendor-derived 1080x2160 timing;
- using generic MIPI packets for manufacturer commands at or above `0xb0`, and
  DCS packets below `0xb0`;
- preserving the display overdrive relay path; and
- `fbcon=rotate:3 fbcon=font:TER16x32` for a usable console orientation and
  font.

The D-PHY timing words reported by the reference implementation are:

| Register | Value |
| --- | --- |
| `TIMCON0` | `0x0a12080a` |
| `TIMCON1` | `0x14320f28` |
| `TIMCON2` | `0x0d320100` |
| `TIMCON3` | `0x00141208` |

These are valuable MT6797/Gemini LK handoff evidence at the reference unit's
link rate. They should enter the current driver first as an explicit validated
test profile with a provenance note, not as an unqualified all-rate or
all-panel constant.

### Immediate relevance to Candidate E

The reference boot history identifies a narrower failure that occurs before a
native panel port is necessary: Linux's unused-clock cleanup disabled
`CLK_INFRA_DISP_PWM` and its `pwm_sel` parent, extinguishing the backlight that
LK had left active. Candidate E's simple-framebuffer node names no clocks and
has no PWM/backlight consumer, so it does not retain that gate.

Linux 7.1.3's simplefb driver enables each clock listed in the framebuffer
node for the node's lifetime. The smallest next A/B candidate is therefore a
DT-only derivative of Candidate E which adds exactly:

```dts
clocks = <&infrasys CLK_INFRA_DISP_PWM>;
```

to `/chosen/framebuffer@7dfb0000`. In the current Linux clock tree that gate is
parented by `pwm_sel`, so enabling the gate also retains the mux parent. This
test should preserve Candidate E's exact kernel, initramfs, framebuffer
geometry and marker. It must not enable or program the native panel, DSI, PWM,
regulators or display power domain.

This is a discriminator, not a conclusion that all scanout clocks are known.
If the backlight is retained but the marker remains absent, a second isolated
clock or native-DRM experiment can follow. Do not combine the PWM retention
property with a broad `clk_ignore_unused` change.

The proposal was checked against the exact Linux 7.1.3 source and Candidate E
package in the development VM:

- `drivers/video/fbdev/simplefb.c` enumerates every clock in the simplefb node
  with `of_clk_get()` and calls `clk_prepare_enable()` on each for the driver's
  lifetime;
- `drivers/clk/mediatek/clk-mt6797.c` defines `infra_disp_pwm` with parent
  `pwm_sel`;
- Candidate E has `CONFIG_FB_SIMPLE=y` and
  `CONFIG_COMMON_CLK_MT6797=y`, while native DRM and the MT6797 MMSYS provider
  remain disabled; and
- its compiled DTB contains `/syscon@10001000` as a one-cell
  `mediatek,mt6797-infracfg` clock provider with an existing phandle.

That compiled DTB does **not** contain `/__symbols__`. Consequently, the
snippet above is the human-readable DTS contract, but the current overlay
cannot rely on an external `&infrasys` fixup. Candidate F's builder must derive
the provider's existing phandle from `/syscon@10001000` in the exact pinned
base DTB (or generate a symbol-bearing combined DTB), reject any provider or
cell-count mismatch, and validate the final property. It must not hard-code
the currently observed numeric phandle.

## Comparison with the Linux 7.1.3 series

| Reference finding | Current 7.1.3 state | Decision |
| --- | --- | --- |
| MT6797 mutex module map and SOF/EOF fields | Patch `0029` already supplies a dedicated MT6797 map, both SOF and EOF fields, and the no-mutex-clock path | Retain and test; do not duplicate the reference patch |
| Overlay IRQ inherited from LK must be cleared before IRQ registration | No corresponding pre-IRQ quiesce exists | Port as a small, reviewable MediaTek DRM patch |
| CRTC interrupt can arrive before `crtc->state` exists | No equivalent early-state guard exists | Port with the overlay handoff fix or as an adjacent logical patch |
| DSI reset must be followed by command-mode selection before the first host transfer | Patch `0038` adds MT6797 host data, but the power-on path does not force command mode after reset | Port as an MT6797 handoff fix, then test independently |
| Hardware-matching D-PHY timing words | The current DSI host uses calculated upstream timing and has no MT6797 override | Add a provenance-marked MT6797/Gemini test profile; keep the generic formula elsewhere and do not assume the words are independent of link rate |
| Correct MT6797 MIPI TX field layout and sequence | Patch `0039` implements MT6797, but comparison found different S2Q divider and bandgap bit definitions and a different prepare/unprepare sequence | Reconcile the local implementation against the hardware-tested fields before enabling the display consumer |
| OD relay must survive dither setup | Patch `0036` skips MT6797's integrated OD dither and keeps OD in relay | Equivalent objective already covered; test rather than stack a second workaround |
| Commands `>= 0xb0` use generic packets | Patch `0043` already applies this split to the provisional Gemini NT36672 table | Retain it in each evidenced Gemini descriptor; do not globalize the threshold to unrelated panels |
| Native DRM owns MMSYS clocks, power domains, PHY, panel, bias and backlight | Candidate E deliberately disables DRM, MMSYS, SCPSYS/genpd and the DSI PHY and uses simplefb | Create a separate native-display diagnostic profile; do not silently broaden Candidate E |
| Working display PWM uses the upstream driver's two named clocks | Local patch `0044` makes `mm` optional and models only `CLK_INFRA_DISP_PWM` as `main` | Re-audit the native PWM contract; the reference DTS uses `CLK_TOP_MUX_PWM` as `main` and `CLK_INFRA_DISP_PWM` as `mm` |
| SSD2092, AUO module, TPS65132-compatible bias path and exact video timings | Directly observed on the reference author's unit; the user's Gemian runtime instead selects an NT36672-named LCM, whose `compare_id()` is unconditional | Do not enable the SSD2092 or NT36672 consumer until a bounded panel-ID or SEEPROM probe resolves the named unit's identity |

One concrete mismatch deserves correction before a native-display build: the
reference implementation places the MT6797 MIPI TX S2Q divider in bits 14:13,
while the current local patch places it in bits 13:12. The reference bandgap
clock/core/low-pass-filter bits also differ from the local definitions. Since
the later reference path reached a stable physical console, its layout is
stronger runtime evidence than the current source-only inference, but the
change still needs a register-level review rather than a blind patch copy.

## Panel identity boundary

The commit's successful panel path is specific to the reference unit:

- direct reference evidence names an AUO/Solomon SSD2092 module;
- the reported mode is 1080x2160, four-lane RGB888 video mode;
- its horizontal timing is HFP 26, HSA 4, HBP 20;
- its vertical timing is VFP 76, VSA 1, VBP 43; and
- its reported pixel clock is 167333 kHz.

The user's named device has the same broad geometry and video-mode class but
Gemian selects `aeon_nt36672_fhd_dsi_vdo_x600_xinli`. That runtime name is not
silicon proof because the vendor comparison routine accepts the panel
unconditionally. The SSD2092 initialization table, reset timings and bias
sequence therefore remain alternate-variant evidence, not safe defaults for
this device. See `bsg100-panel-crosscheck-20260714.txt` for the full conflict.

## Prioritized port and test plan

1. The one-variable simplefb backlight-clock retention A/B has now run.
   Candidate F produced about one second of sideways fbcon text before black,
   the first positive visual Linux 7.1.3 signal on this unit. Candidate G kept
   the exact F kernel/DTB, removed only the raw initramfs overwrite, and
   reproduced sideways scrolling for 1–2 seconds before black. Candidate H
   kept G's kernel/initramfs and added only MM-root clock retention. Two
   attempts visibly progressed farther and approximately exposed its
   initramfs-only marker before the screen and backlight went off; later
   attempts did not reproduce the progress. Candidate I's exact-H
   initramfs-only counter follow-up then went directly to black without its
   marker, leaving selection and `/init` unconfirmed. Candidate J used the same
   DTB/initramfs/container command line with a rebuilt kernel whose forced
   command line adds broad `clk_ignore_unused` retention. Its runtime reports
   provisionally describe three intended selections: two tick-04-compatible
   visible-to-black outcomes and one direct-black/no-console outcome that
   cannot establish kernel or `/init` execution. Stable visibility and clock
   causality remain unestablished. Stop further J repetition. The completed
   reassessment selected Candidate K, an exact-J initramfs-only derivative that
   separates 20 fixed-width CR/no-newline updates from 12 controlled newline
   lines. Asynchronous printk remains a confounder. K is synchronized and fully
   read back from `boot2`; one attended K selection is the next gate.
2. Port the generic LK-to-Linux handoff fixes: overlay IRQ quiesce, early CRTC
   state guard, and DSI command-mode selection after reset.
3. Reconcile the MT6797 MIPI TX register fields and move sleepable setup into
   the PHY prepare/unprepare path where appropriate.
4. Add the hardware-tested D-PHY timing words as a provenance-marked
   MT6797/Gemini test profile, retaining calculated timing for other rates and
   platforms.
5. Preserve the already-equivalent mutex, OD-relay and packet-type work rather
   than duplicating it.
6. Resolve panel identity on the user's unit with a bounded read-only probe.
   Select the SSD2092 path only if the direct result matches; otherwise use the
   confirmed controller and its own exact initialization/mode data.
7. Build a separate, conservative native-display diagnostic candidate with all
   required display dependencies built in, one CPU, storage writes kept inert,
   fbcon rotation/font parameters, and an independent boot-stage marker because
   UART is unavailable.
8. First boot only from the known selectable non-primary slot and record panel,
   backlight, timeout, power and recovery observations before changing another
   variable.

## Conclusion

This commit is the best current starting point for the native MT6797 DRM path,
not a drop-in boot image or a proof of the user's panel model. The portable
MediaTek handoff and PHY fixes should be incorporated first. Panel-specific
code must remain gated by direct identity evidence from the named device.

## Local runtime closure

The [Candidate F result](../../2026-07-16-screen-clock-retention-diagnostic/results/runtime-candidate-f-20260716.txt)
supports this audit's clock-retention lead: with exact Candidate E payloads and
only the simplefb `CLK_INFRA_DISP_PWM` reference added, sideways fbcon text was
visible for about one second. It does not prove native DRM or this reference's
panel data. The [Candidate G experiment](../../2026-07-16-fbcon-text-diagnostic/README.md)
removes Candidate F's raw framebuffer overwrite without changing its kernel or
DTB. G reproduced sideways output before the display again went black, rejecting
the overwrite explanation. The [Candidate H experiment](../../2026-07-16-simplefb-mm-root-retention/README.md)
isolated MM-root retention but did not make visible output repeatable. Its
approximately recognized initramfs-only marker strongly attributes two visible
attempts to external `/init`. The [Candidate I
experiment](../../2026-07-16-fbcon-refresh-timing-diagnostic/README.md) keeps
H's exact kernel/DTB and changes only `/init` to time the black transition
against one tty0 line per second and a subsequent silent hold. Its reported
intended selection went directly to black without the I marker or counter, so
that timing hypothesis remains untested. The [Candidate J experiment](../../2026-07-17-clk-ignore-unused-diagnostic/README.md)
was synchronized and fully read back from `boot2`; it rebuilds the kernel
with `clk_ignore_unused` in forced `CONFIG_CMDLINE` while preserving I's exact
DTB, initramfs, and Android header command line. Its first intended selection
showed the reported shared `/init` suffix `4/60` before black. A later
two-bullet report is provisionally interpreted as two additional intended
selections because the outcomes are mutually exclusive, with owner
confirmation pending. One reached "iteration 4" before black, compatible with
and corroborating tick 04, while one went directly black with no console and
cannot establish kernel or `/init` execution. Provisionally, two of three
intended selections had tick-04-compatible visible output, but stable
visibility and clock causality remain unestablished. Further J repetition is
stopped; no matched-I rollback is authorized. The completed reassessment
selected the [Candidate K experiment](../../2026-07-17-fbcon-newline-boundary-diagnostic/README.md),
which retains exact J's kernel/DTB/container layout and changes only `/init`.
Its 20 fixed-width CR/no-newline updates precede 12 controlled newline lines;
asynchronous printk remains a confounder. K is synchronized and fully read back
from `boot2`; runtime is pending. The exact kernel has fbcon rotation compiled
out, so the bsg100-tested `fbcon=rotate:3` behavior remains a later separate
configuration delta.
