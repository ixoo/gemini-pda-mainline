## Question

The owner reports are provisionally mapped to three intended Candidate J
`boot2` selections, with confirmation of the later attempt count and order
pending. The first reported `4/60`, strongly attributed to the tracked `04/60`
tick; one later bullet is compatible with the same tick-04 boundary, while the
other reports no console. Does this recurring tick-04-compatible boundary
depend on newline and scroll activity generated by Candidate J's
one-line-per-second program, or does the display disappear at about the same
elapsed point when `/init` rewrites the current row without deliberate
newlines?

Candidate K is a deliberately narrow initramfs-only derivative of exact
Candidate J. It does not identify a clock, panel or backlight cause. It tests
only whether the visible boundary follows newline/scroll behavior deliberately
generated by `/init`. Candidate J's forced `ignore_loglevel loglevel=8` remains
unchanged, so asynchronous kernel printk output can still add console newlines
and is an explicit confounder.

## Runtime program

After mounting devtmpfs, read-only procfs and read-only sysfs, tracked `/init`
writes to `/dev/tty0` and to `/dev/ttyS0` when each character device exists.
It has three phases:

1. Wait one second, then rewrite one fixed-width 54-character line with a
   leading carriage return and **no newline**, exactly 20 times:

   ```text
   GEMINI_FBCON_BOUNDARY_20260717_K PHASE1 CR-NO-NL 01/20
   ...
   GEMINI_FBCON_BOUNDARY_20260717_K PHASE1 CR-NO-NL 20/20
   ```

2. Emit a distinct newline-terminated transition, then wait one second before
   each of exactly 12 newline-terminated lines:

   ```text
   GEMINI_FBCON_BOUNDARY_20260717_K PHASE2 NEWLINE BEGIN 00/12
   GEMINI_FBCON_BOUNDARY_20260717_K PHASE2 NEWLINE 01/12
   ...
   GEMINI_FBCON_BOUNDARY_20260717_K PHASE2 NEWLINE 12/12
   ```

3. Emit `GEMINI_FBCON_BOUNDARY_20260717_K STATIC HOLD; NO FURTHER
   CONSOLE WRITES`, then sleep indefinitely without another deliberate console
   write.

The leading carriage return makes every phase-1 update begin at column zero.
The two-digit counter keeps the line width constant, and the validator enforces
a conservative maximum of 72 columns to avoid an implicit wrap on a narrow
console. Each counter is emitted only after its associated one-second sleep
and increment.

## Safety boundary

The builder is file-only and has no partition or device interface. After
mounting devtmpfs plus read-only procfs and sysfs, the only writable
hardware-facing device nodes used by tracked Candidate K `/init` are
`/dev/tty0` and `/dev/ttyS0`; diagnostics are redirected to `/dev/null`. It does
not explicitly access eMMC, an fbdev node, raw memory, MMIO, I2C, watchdog,
reset/power, network or USB control interfaces; run `dd`; program a clock or
regulator; request reboot; or deliberately write persistent state. This claim
is scoped to the tracked `/init`, not the exact-J kernel it runs: that kernel
retains its normal MMIO, timer, clock, console and fbcon behavior. Dormant
initramfs binaries inherited from exact J remain byte-identical and are not
invoked by `/init`.

The broad diagnostic `clk_ignore_unused` kernel option from Candidate J remains
present because changing it would add a second variable. It is not a normal
boot policy and does not enable already-off clocks, prevent explicit disables,
or retain regulators and power domains. Any later device synchronization is a
separate operation under the standing logical-`boot2` policy in `AGENTS.md`.
Stop under the repository's heat, battery, filesystem, watchdog-loop or
recovery-change conditions.

The later standing-authorized synchronization is separately recorded in
[`results/boot2-write-candidate-k-20260717.txt`](results/boot2-write-candidate-k-20260717.txt).
It resolved the live GPT label, preserved exact Candidate J, wrote only
logical `boot2`, synchronized and block-flushed it, and required a
byte-identical full local readback. It did not reboot or shut down the device.

## Attended test and interpretation

Perform one attended logical-`boot2` selection after returning through the
known-good OS and allowing the device to reach a normal temperature. Video is
preferred. Record the exact first and last visible marker/counter, black-screen
time, whether pixels and backlight disappear together, apparent power/reboot
state and recovery action. An attempt without the unique K marker does not
confirm Candidate K `/init` execution. End the observation after phase 2
(approximately 32 seconds after the first loop begins), return to the known-good
OS, and do not leave K in its indefinite static hold unattended because it
retains Candidate J's broad `clk_ignore_unused` diagnostic.

- Black near `04/20` during phase 1 rejects `/init`-generated newline/scroll
  activity as a sufficient explanation for the tick-04-like boundary. It does
  not exclude asynchronous kernel-console activity or identify the actual
  display or power cause.
- Visibility through `20/20`, followed by a transition near a phase-2 newline
  counter, supports a newline/scroll-sensitive effect. Repetition would still
  be required, and correlation would not identify a driver or clock.
- Visibility through `12/12` followed by black only during static hold supports
  an activity-dependent retention effect rather than a newline-count boundary.
- A partial marker without a readable counter establishes only that the first
  post-sleep phase-1 emit began; it does not bound the black transition more
  precisely. No marker remains unattributable.
