Register evidence and compatibility decision
---------------------------------------------

The live snapshot reproduces the MT6797 register contract recovered from the
Planet source. `PATCH_BIT1=0xfffe00c0` has bit 7 set; `PATCH_BIT2=0xa48d180d`
has the support-64G bit clear; DMA high-address register `0x8c` is zero; and
`PAD_TUNE0=0x002b2100` is active while `PAD_TUNE1=0`. The vendor map defines a
12-bit clock divider, asynchronous FIFO/data tuning, and `0xf0` PAD_TUNE0.
Register `0x228` is `EMMC50_BLOCK_LENGTH`, not the newer `SDC_FIFO_CFG`, and
the map has no `SDC_ADV_CFG0` at `0x64`.

Linux 7.1.3's existing `mtk-sd` driver therefore remains the correct driver;
the dedicated `mt6797_compat` record is required and `mt6779_compat` is not a
safe substitute:

  * `clk_div_bits=12`, `pad_tune_reg=MSDC_PAD_TUNE0`, `async_fifo=true`, and
    `data_tune=true`;
  * `busy_check=false`, `stop_clk_fix=false`, and `enhance_rx=false`;
  * `support_64g=false` until descriptor high-address use is proven. The
    downstream implementation truncates descriptor/payload addresses and the
    live high-address register is zero.

Mainline package and board boundary
-----------------------------------

The current package links `CONFIG_MMC=y`, `CONFIG_MMC_MTK=y`, and
`CONFIG_MTK_PMIC_WRAP=y`; `System.map` contains `msdc_drv_probe`,
`msdc_ops_set_ios`, `msdc_ops_switch_volt`, and the PMIC wrapper/regulator
probes. The Gemini DTB contains:

  * `mmc@11230000`, `mediatek,mt6797-mmc`, SPI 79, `source`/`hclk`/`source_cg`
    clocks, status okay, 8-bit, `max-frequency = 25 MHz`, `vmmc = vemc_3v3`,
    `vqmmc = vio18`, `non-removable`, `no-sd`, and `no-sdio`;
  * identical pinmux-only default and `state_uhs` groups, with all generic
    input-enable/drive/bias properties removed by patch 0071;
  * `mmc@11240000`, SPI 80 and the MT6797 compatibility data, disabled.

The source-only probe audit confirms that `msdc_drv_probe()` enables clocks,
allocates DMA descriptors, resets/programs FIFO/interrupt/DMA/tuning/timeout
registers, requests the IRQ, enables runtime PM, and calls `mmc_add_host()`.
MMC core can then power up the eMMC, enable VEMC/VIO18, identify the card, and
later change VQMMC/pinctrl during a signal-voltage switch. This is a stateful
probe, not a read-only check. The first runtime attempt must use an external
or non-primary recovery path and a read-only rootfs/storage policy.

The MT6797 pinctrl descriptor exposes mode, direction, data-in, and data-out
register maps only; no source-backed pull, IES, or drive callbacks exist.
The pinmux-only board correction is therefore retained. High-speed eMMC,
MSDC1/card-detect, UHS voltage switching, and 64G descriptor mode remain
disabled until controlled runtime evidence exists.

Cross-repository comparison
---------------------------

The early [bsg100/gemini-linux](https://github.com/bsg100/gemini-linux) report
and our pinned [comparison audit](../../2026-07-13-bsg100-gemini-linux-comparison/README.md)
are consistent with the current source audit where they say generic pinconf
properties prevented the first eMMC boot:
the current 7.1.3 MT6797 descriptor still has the same four register maps and
no generic pinconf callbacks. It is treated as a hypothesis and compatibility
clue, not as proof of pad values or a reason to copy the old vendor tree.
